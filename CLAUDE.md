## Project Overview

**AGRON** is an AI-powered crop health companion platform. The repo is a monorepo containing:
- A React Native/Expo mobile + web app (`agrof-main/mobile/app/`)
- A **merchant portal website** (`portal/`) ŌĆö Next.js 14, for traders to manage stores & chat
- An AWS Amplify Gen 2 backend (`amplify/`)
- A Python Flask AI analysis API (`agrof-main/src/api/`)
- A Node.js Express store catalog backend (`store-backend/`)

Firebase and Supabase have been **fully removed**. The sole backend is AWS Amplify (Cognito, AppSync, S3).

## Commands

### Local Development (run each in a separate terminal)
```bash
npm run sandbox              # Start AWS Amplify local backend, generates amplify_outputs.json
npm run amplify:copy-outputs # Copy amplify_outputs.json into the Expo app
npm start                    # Start Expo dev server (port 8086)
npm run start:store          # Start store backend (Node/Express, port 3001)
npm run start:backend        # Start Python Flask AI API (port 5000)
```

### Portal (from `portal/`)
```bash
npm install     # First time ŌĆö install Next.js dependencies
npm run dev     # Portal dev server (http://localhost:3000)
npm run build   # Production build
```

### Mobile App (from `agrof-main/mobile/app/`)
```bash
npm start         # Metro/Expo dev server
npm run android   # Run on Android
npm run ios       # Run on iOS
npm run build:web # Export Expo web app to /dist
```

### Python Backend (from `agrof-main/` via Makefile)

make setup    # Create venv and install Python deps

## Architecture

### Two Frontends, One Backend

**Expo app** (`agrof-main/mobile/app/`) ŌĆö mobile (Android/iOS) and web (Expo for Web). Primary product for farmers: AI disease detection, store browsing, P2P market.

**Merchant portal** (`portal/`) ŌĆö Next.js 14 (App Router) website for traders. Routes:
- `/` ŌĆö landing page
- `/login`, `/signup` ŌĆö Cognito auth
- `/dashboard` ŌĆö overview stats
- `/dashboard/store` ŌĆö manage own MarketListing records
- `/dashboard/market` ŌĆö browse all active listings, message sellers
- `/dashboard/messages/[chatId]` ŌĆö encrypted 1:1 chat

Both share the same Amplify backend (Cognito user pool, AppSync GraphQL, S3).

### Auth: AWS Cognito (Amplify Gen 2)
`amplify/auth/resource.ts` defines the user pool (email-based, code verification). The Expo app initialises Amplify in `index.js` using `amplify_outputs.json`. The portal initialises in `portal/components/AmplifyProvider.tsx` via `portal/lib/amplify.ts`. `AUTH_PROVIDER` in `authConfig.js` is hardcoded to `'cognito'`.

### Backend Config: `amplify_outputs.json`
Generated by `npm run sandbox` (not committed). Without it, auth will not initialise. Must be copied into `agrof-main/mobile/app/` before running Expo. The portal reads it via a symlink at `portal/amplify_outputs.json`.

### AppSync GraphQL Schema (`amplify/data/resource.ts`)
Five models:
- `SyncedRecord` ŌĆö generic JSON for TanStack Query + SQLite sync bridge
- `StoreListing` ŌĆö legacy Expo app marketplace listing (kept for compat)
- `MerchantProfile` ŌĆö merchant display name, role (BUYER/SELLER/BOTH), and TweetNaCl X25519 **public key** for E2E encryption
- `MarketListing` ŌĆö portal P2P listings (title, category, price, unit, S3 image keys, status)
- `ChatMessage` ŌĆö encrypted message: `chatId`, `fromSub`, `toSub`, `ciphertext` (base64), `nonce` (base64)

### End-to-End Encrypted Chat
On first portal login, `AuthGuard.tsx` generates a TweetNaCl X25519 keypair. The public key is stored in `MerchantProfile` (AppSync). The private key is stored in `localStorage`. Messages are encrypted with `nacl.box()` using the recipient's public key before being saved to AppSync ŌĆö the server only ever stores ciphertext. See `portal/lib/crypto.ts` for encrypt/decrypt helpers.

### Dynamic API Endpoint Discovery
On startup, `apiConfig.js` probes a list of LAN IPs on ports 3001 (store) and 5000 (AI API) to find a working endpoint. Can cause slow startup if unreachable. Set `EXPO_PUBLIC_DISABLE_LAN_API_PROBE=1` to skip.

### Local-First Data: SQLite + TanStack Query
Expo app uses Expo SQLite for local persistence and TanStack Query for server state. `lib/sqlite/database.js` manages the local DB. `services/syncBridge.js` syncs local SQLite to AppSync `SyncedRecord`. Portal uses TanStack Query directly against AppSync.

### Store API Routing ŌĆö Portal as the Official Store API

The portal (Next.js) is the **only** public-facing API for the store. The mobile app and any other client must call the portal, not the store-backend directly.

```
Mobile App  ŌåÆ  portal /api/store/*  ŌåÆ  store-backend Express (internal, port 3001)
Portal UI   ŌåÆ  portal /api/admin/*  ŌåÆ  store-backend Express (internal, port 3001)
```

**Portal store routes** (`portal/app/api/store/[...path]/route.ts`): catch-all proxy, forwards to `STORE_BACKEND_URL`. No auth required ŌĆö these are read endpoints for browsing.

**Portal admin routes** (`portal/app/api/admin/`): protected by Cognito JWT (`Authorization: Bearer <idToken>`). Used by the portal catalog UI. Calls store-backend for CRUD.

**Store-backend** (`store-backend/server.js`): internal service only. Exposes full product API + new `POST/PUT/DELETE /api/products` endpoints for the portal admin.

**Mobile app config** (`apiConfig.js`): when `EXPO_PUBLIC_STORE_PORTAL_URL` is set, `API_CONFIG.STORE.API_URL` points to `${PORTAL_URL}/api/store`. Without it, falls back to LAN discovery (local dev only).

### Merchant Catalog UI (portal)
- `/dashboard/catalog` ŌĆö browse, search, filter, discontinue products
- `/dashboard/catalog/new` ŌĆö add new product with full details + pricing
- `/dashboard/catalog/[id]/edit` ŌĆö edit any product field
- `/dashboard/catalog/[id]/stock` ŌĆö stock adjustments (IN/OUT/ADJUSTMENT) + transaction history
- `/dashboard/inventory` ŌĆö stock alerts, out-of-stock list, low-stock table, analytics

### Store Catalog vs MarketListing
- **SQLite store-backend** (Express, port 3001): authoritative product catalog (1000+ products, 6 categories, inventory). Managed via portal admin UI. Served to mobile app through portal `/api/store/*`.
- **AppSync `MarketListing`**: individual trader P2P offers posted directly by merchants. Portal reads/writes via AppSync. Separate from the main catalog.

### Internationalisation
i18next with locales in `agrof-main/mobile/app/locales/` (en, lg, rn ŌĆö English, Luganda, Kirundi). Language state managed by `LanguageContext.js`.

### Mobile App Navigation
Tab-based navigation driven by `currentTab` state in `App.js`. No top-level React Navigation stack ŌĆö navigation is controlled via state. `AuthGate.js` wraps protected content. `P2PMarketPanel.js` contains a portal deeplink banner that opens `EXPO_PUBLIC_PORTAL_URL` (default: `https://portal.agron.uk`).

## Deployment (AWS Amplify Hosting)

Defined in `amplify.yml` ŌĆö two app targets sharing one backend deployment:

1. **Expo app** (`appRoot: agrof-main/mobile/app`): backend deploy via `npx ampx pipeline-deploy`, then `npm run build:web` ŌåÆ artifact at `dist/`
2. **Portal** (`appRoot: portal`): `npm run build` ŌåÆ artifact at `.next/`

Configure each as a separate Amplify Hosting app in the Console, connected to this repo on the same branch. The portal's Next.js output uses `standalone` mode.

## Gotchas

- **Cognito USER_PASSWORD_AUTH** is explicitly enabled on the Cognito app client to support Expo Go. Do not disable it.
- **Gemini API key** is hardcoded in `agrof-main/src/api/app.py`. Move to env var before production.
- **Email verification** requires the sender domain (`agron.uk`) verified in Amazon SES in `us-east-1`.
- **Portal private key**: TweetNaCl private keys are in `localStorage` under `agron_portal_sk`. If a user clears localStorage, their key is lost and old messages become unreadable. Future improvement: key backup via Cognito user attributes.
- **Portal amplify_outputs.json** is a symlink to the root file (`portal/amplify_outputs.json ŌåÆ ../amplify_outputs.json`). Run `npm run sandbox` at root before `npm run dev` in the portal.
- The `store/` directory at root is a catalog JSON template ŌĆö not used at runtime.
